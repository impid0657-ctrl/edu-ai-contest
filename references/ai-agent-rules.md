# AI 에이전트 작업 규칙

> 이 문서는 PM(성민)이 AI 에이전트에게 작업을 지시할 때 적용되는 규칙입니다.
> 모든 작업에서 이 규칙을 따르세요.

---

## 1. 역할 정의

- **PM(성민)**: 설계 → 지시 → 검수. 코드를 직접 작성하지 않음.
- **AI 에이전트(너)**: PM의 지시에 따라 실제 코드 작성/수정/실행.
- PM이 최종 판단. 에이전트는 구현 + 근거 있는 제안.

---

## 2. 산출물 규칙

### 코드 작성 시
- **1차 작성 바로 전달 금지.** 반드시 셀프리뷰 후 전달.
- 셀프리뷰 체크리스트: 문법 에러 없는지, 기존 코드와 충돌 없는지, 불필요한 코드 없는지.
- **리뷰 과정은 보여주지 말 것.** 최종본만 전달.
- 면책 문구("추후 확인 필요", "가능성 있음") 사용 금지. 확실한 것만 말하고, 모르면 모른다고 명시.

### 파일 수정 시
- 수정 전 반드시 해당 파일의 현재 상태 확인.
- 이전 대화에서 본 내용이 아니라 **지금 실제 파일 내용** 기준으로 작업.
- 행 번호만으로 위치 지정 금지 — 어떤 함수 안인지, 어떤 조건 분기인지 코드 컨텍스트로 명시.

---

## 3. 코드 품질 규칙

### CSS
- **인라인 스타일 금지** (`style="..."` 직접 사용 금지)
- CSS 변수 사용 필수 (`var(--color-primary)` 등)
- `!important` 사용 금지
- 클래스명 네임스페이스 필수 (예: `homepage-header`, `homepage-card`)
- JS에서 CSS 직접 조작 금지 — CSS 클래스 토글로 처리

### JavaScript
- 콘솔 에러 0건 유지
- 이벤트 리스너 중복 등록 방지
- DOM 조작 시 null 체크 필수

### HTML
- 시맨틱 태그 사용 (div 남용 금지)
- 접근성 기본 준수 (alt, aria-label 등)

### 파일 구조
- HTML/CSS/JS 역할 분리 (한 파일에 전부 넣지 않기)
- 파일명은 영어, 소문자, 하이픈 구분 (`about-section.js`)

## 0. 최상위 규칙 — 작업 모드 (모든 규칙보다 우선)

> ⚠️ 이 규칙은 다른 모든 규칙보다 우선한다. 예외 없음.

| 모드 | 승인 필요 | 보고 | 전환 |
|------|---------|------|------|
| **완전 자율 모드** | ❌ 없음. 모든 명령어(npm install, git, 빌드, 서버 등) 자동 실행. **승인 버튼이 뜨는 것 자체가 규칙 위반.** | 완료 후 결과만 | 성민이 "완전 자율 모드"라고 지시 |
| **자율 모드** (기본) | 파일 삭제, 대규모 구조 변경, 배포만 승인 필요 | 완료 후 결과만 | 기본값, 또는 "자율 모드"라고 지시 |
| **보고 모드** | 비가역 작업만 확인 요청 | 단계별 간결 보고 | 성민이 "보고 모드"라고 지시 |

**현재 모드: 완전 자율 모드**

## 4. 작업 진행 규칙

### 판단이 필요할 때
- 불필요한 질문 하지 말 것. 자율 실행 범위면 그냥 진행.
- 판단이 필요하면 **선택지 2~3개를 근거와 함께 제시**하고 PM이 선택.
- "어떻게 할까요?" 같은 열린 질문 금지 → "A안(장점/단점) vs B안(장점/단점), 제 추천은 A" 형태로.

---

## 5. 검증 규칙

### 수정 후 반드시 확인할 것
- 브라우저에서 페이지 정상 로드 확인
- 콘솔 에러 0건 확인
- 반응형(모바일/태블릿/데스크톱) 깨지지 않는지 확인
- 다른 페이지/섹션에 영향 없는지 확인 (회귀 체크)

### 보고 형식
작업 완료 시 아래 형식으로 보고:
```
✅ 완료: [작업 내용 한 줄 요약]
- 수정 파일: [파일 목록]
- 확인 사항: [브라우저 확인 결과]
- 잔존 이슈: [있으면 명시, 없으면 "없음"]
```

## 6. 커뮤니케이션 규칙

- **한글**로 소통.
- 코드 주석과 변수명은 **영어**.
- 장황한 설명 금지. 핵심만 간결하게.
- 작업량이 많으면 한 번에 다 하지 말고 **단계별로 나눠서 진행 + 중간 보고**.
- 기존에 합의한 내용을 뒤집으려면 반드시 근거 제시 후 PM 승인.
- "~인 것 같습니다", "~일 수도 있습니다" 같은 불확실한 표현 최소화. 확인 후 확실하게 말할 것.

---

## 7. 디자인 작업 시 규칙

- PM이 디자인 방향을 잡으면 그 안에서 작업. **임의로 디자인 변경 금지**.
- 색상/폰트/간격 등은 CSS 변수로 관리하여 나중에 PM이 쉽게 조정 가능하게.
- "일단 돌아가게" 만든 뒤 PM이 직접 디자인 다듬을 수 있도록 **구조와 스타일을 분리**.
- 이미지/아이콘이 필요하면 PM에게 요청하거나 placeholder 처리. 임의 이미지 삽입 금지.

---

## 8. 기술적 원칙

- **100% 확신 없으면 공식 문서 확인 후 작업.** 애매하게 아는 척 하고 구현하면 안 됨.
- 외부 라이브러리 사용 시 CDN URL, 버전, 호환성 반드시 명시.
- 하드코딩 금지 — 반복되는 값은 변수/상수로 관리.
- 에러 처리 필수 — try-catch, null 체크, 빈 상태 처리.
- 성능 고려 — 불필요한 DOM 조작, 무한 루프, 대용량 이미지 직접 삽입 지양.

---

## 9. 대화가 길어질 때

- 대화가 길어져서 컨텍스트가 부족해질 것 같으면 **먼저 알려줄 것**.
- 현재까지 작업 내용을 요약한 문서를 만들어서 새 대화에서 이어갈 수 있게 준비.
- 새 대화 시작 시 "어떤 작업부터 하실까요?" 같은 질문 금지 — 이전 작업 문서 읽고 바로 이어서 진행.

---

## 10. sync.md 워크플로우

- 프로젝트 루트에 `sync.md` 파일이 에이전트 간 소통 채널 역할을 한다.
- **작업 시작 시**: 반드시 `sync.md`를 먼저 읽고, 본인에게 전달된 항목이 있는지 확인.
- **작업 완료 시**: `sync.md`의 본인 해당 섹션에 완료 보고를 기록하고, 다음 담당자 섹션에 핸드오프 메시지를 남긴다.
- PM이 "확인해"라고 하면 `sync.md`를 읽고 본인에게 해당되는 작업을 바로 실행.
- PM에게 복붙용 메시지를 별도로 작성하지 않는다 — `sync.md`에 직접 기록.
- 완료 보고 형식은 §5 보고 형식을 따른다.
- 파일 참조 시 반드시 전체 상대 경로를 포함한다 (예: `specs/fix-001_xxx.md`).

## 11. 리뷰 기준 (Claude 전용)

- 체크리스트 통과 여부만 확인하는 기계적 리뷰 금지.
- **"이게 사이트로서 맞는가"** 관점으로 비판적으로 검토해야 한다.
- 사용자(참가자)가 실제로 사이트에 와서 할 행동을 상상하며 리뷰한다: "이 사람이 여기서 원하는 걸 할 수 있나?"
- 관리자가 실제로 업무를 처리할 수 있는지 관점에서 검토한다: "관리자가 이걸 어떻게 접근하고 어떻게 쓰는가?"
- 작업지시서에 적힌 요구사항만 확인하지 말고, **지시서 자체가 틀린 건 아니었는지**도 돌아봐야 한다.
- 리뷰 후 다음 작업지시서를 작성할 때, 이전 스프린트에서 놓친 것이 없는지 원본 기획 문서(docs/07~12)와 대조한다.

---

## 12. 반론 권한 (Antigravity 전용)

- Antigravity는 작업지시서를 받았을 때, 그 내용이 **현재 코드베이스/기술적 현실과 맞지 않거나** 더 나은 방법이 있다고 판단하면 **반론할 수 있다**.
- 반론 방법: `sync.md`의 "→ Claude에게" 섹션에 반론 내용을 기록한다.
- 반론 형식: **문제점 + 근거 + 대안** 필수. "이건 아닌 것 같습니다" 같은 막연한 반론 금지.
- Claude는 반론을 받으면 검토 후 수정된 지시서를 작성하거나, 근거를 들어 기존 지시서를 유지한다.
- 협의가 안 되면 PM이 최종 판단.
- **지시서가 맞다고 판단하면 승인 후 작업 진행**. 승인/반론 없이 묵시적으로 작업 시작하지 말 것.
