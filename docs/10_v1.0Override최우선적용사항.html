10.1 대용량 파일 저장소 전략

문제 상황

Supabase Storage의 무료 플랜은 1GB 제한이 있으며, 팀당 최대 500MB 파일을 허용하는 경우 약 2개 팀만 접수 가능합니다. 목표 500팀 접수 시 최대 250GB가 필요합니다.

해결 방안

전략 내용 비용 및 평가
방안 1: Supabase Pro 플랜 100GB 기본 제공 + 초과분 GB당 $0.021 ✅ 권장 (통합 관리 용이)
예상 사용량 250GB 시 약 $28/월
방안 2: AWS S3 별도 사용 S3 Standard 250GB 저장 시 약 $5.75/월 💰 저렴하나 개발 복잡도 증가
단, Supabase와 별도 연동 코드 필요
방안 3: Cloudflare R2 무료 10GB + 초과분 GB당 $0.015
Egress(다운로드) 무료		 ✅ 가격 경쟁력 있음

최종 권장 사항

  • Supabase Pro 플랜 ($25/월)을 사용하되, 다음 최적화 전략 병행:
  • 접수 마감 후 불필요한 파일 아카이빙(압축) 처리
  • 심사 완료 후 3개월 뒤 자동 삭제 정책 적용
  • 참가작 미리보기는 썸네일 생성하여 별도 저장 (용량 절감)

10.2 RLS 정책 수정 (Storage Path)

🔥 [12.6 최종 적용] 비회원 권한 문제 및 해결책
기존 8장 및 10장 초기 버전의 RLS 정책은 auth.uid()를 기반으로 합니다. 하지만 비회원은 auth.uid()가 NULL이므로 브라우저 직접 업로드 시 100% 403 에러가 발생합니다.
따라서 비회원은 반드시 서버 API(/api/guest/upload)를 경유하여 Service Role Key로 업로드해야 합니다.

수정된 RLS 정책 (Path-based RLS)

회원 및 관리자를 위한 RLS 정책은 아래와 같이 경로 기반으로 설정합니다. 비회원은 이 정책을 우회하는 서버 측 업로드를 사용합니다. 

-- 1. 팀별 폴더 구조 사용: submissions/{team_id}/{filename}
create policy "Users can upload to own team folder"
on storage.objects for insert
with check (
  bucket_id = 'submissions' and
  (storage.foldername(name))[1] = auth.uid()::text
);

-- 2. 본인 팀 파일만 조회 가능
create policy "Users can view own team files"
on storage.objects for select
using (
  bucket_id = 'submissions' and
  (storage.foldername(name))[1] = auth.uid()::text
);

-- 3. 관리자는 모든 파일 접근 가능
create policy "Admins can access all files"
on storage.objects for select
using (
  bucket_id = 'submissions' and
  exists (
    select 1 from public.users
    where users.id = auth.uid()
    and users.role = 'admin'
  )
);

10.3 AI 심사 처리 방식

🔥 [12.4 최종 적용] 수동 트리거 강제
기존의 "파일 업로드 시 자동 Webhook 트리거" 방식은 Vercel 타임아웃 문제로 전면 폐기되었습니다.
반드시 관리자 수동 트리거 + 백그라운드 큐(Queue) 방식으로 구현해야 합니다.

비동기 처리 아키텍처

[파일 업로드 완료] → DB ai_review_status = 'pending' 저장 → [관리자 대시보드 클릭] → POST /api/admin/ai-review/trigger → ai_review_queue 테이블 등록 → 백그라운드 워커 처리

관리자 수동 트리거 API 코드

// POST /api/admin/ai-review/trigger
// 관리자 세션 인증 필수

export async function POST(req: Request) {
  // 1. pending 상태의 심사 대상 조회
  const { data: pendingApps } = await supabaseAdmin
    .from('applications')
    .select('id, file_url, application_number')
    .eq('ai_review_status', 'pending');

  if (!pendingApps || pendingApps.length === 0) {
    return Response.json({ message: '처리할 대기 항목이 없습니다.' });
  }

  // 2. ai_review_queue 테이블에 일괄 등록
  const queueItems = pendingApps.map(app => ({
    application_id: app.id,
    file_url: app.file_url,
    status: 'queued',
    created_at: new Date().toISOString(),
  }));

  const { error } = await supabaseAdmin
    .from('ai_review_queue')
    .insert(queueItems);

  if (error) return Response.json({ error }, { status: 500 });

  // 3. applications 테이블 상태를 'queued'로 업데이트
  await supabaseAdmin
    .from('applications')
    .update({ ai_review_status: 'queued' })
    .in('id', pendingApps.map(a => a.id));

  return Response.json({ 
    success: true, 
    queued: pendingApps.length, 
    message: `${pendingApps.length}건의 AI 심사가 큐에 등록되었습니다.` 
  });
}
